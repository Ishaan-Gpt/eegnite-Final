| Feature | 🟢 Production Build | 🟡 Staging Build (Current) |
|---------|-------------------|---------------------------|

   ```

   (No X-Robots-Tag)              Header always set X-Robots-Tag

   ```

3. **Meta Tags** (generated by layout.tsx)
   - Both check environment, but staging domains automatically get noindex tags
   - Production domain (eegnite.com) gets index tags

## How to Create Production Build (When Ready)

When you're ready to deploy to production:

### Step 1: Restore Production robots.txt
```bash
# Don't copy robots-staging.txt
# Use the original robots.txt from public/ folder
# Or regenerate with production settings
```

### Step 2: Remove X-Robots-Tag from .htaccess
Edit `.htaccess` and remove these lines:
```apache
# STAGING: Block all search engine indexing
Header always set X-Robots-Tag "noindex, nofollow, noarchive, nosnippet"
```

### Step 3: Build with Production URL
```bash
# Set environment variable
NEXT_PUBLIC_SITE_URL=https://www.eegnite.com npm run build

# This ensures the layout generates proper meta tags
```

### Step 4: Remove Staging Documentation
Delete from the build:
- `STAGING_README.md`
- `STAGING_INFO.html`

### Step 5: Deploy to Production Domain
Upload to your main domain (eegnite.com)

## Security Layers Explained

### Layer 1: robots.txt (Polite Request)
- Search engines "honor" this file voluntarily
- Not a security measure, but a directive
- Good crawlers will respect it

### Layer 2: Meta Tags (Page-Level Directive)
- More explicit than robots.txt
- Applied to each individual page
- Tells crawlers not to index this specific page

### Layer 3: HTTP Headers (Server-Level Directive)
- Most robust method
- Applied at the server level before HTML is parsed
- Works even if the HTML is malformed
- Includes `noarchive` and `nosnippet` for extra protection

### All 3 Together = Maximum Protection
Using all three ensures that even if one layer fails or is ignored, the others will prevent indexing.

## Common Questions

**Q: Will Google still crawl my staging site?**
A: Google may still discover it and crawl it, but it won't be indexed (won't appear in search results).

**Q: Can I use this for a public preview?**
A: Yes! The site is accessible to anyone with the URL, but search engines won't index it.

**Q: What if someone shares the staging link?**
A: People can still visit it, but it won't show up in Google searches. Consider adding password protection via .htaccess for more security.

**Q: How long until Google removes it from index (if it was previously indexed)?**
A: If the staging domain was previously indexed and you add these protections, Google typically removes it within a few weeks.

**Q: Should I also add password protection?**
A: For sensitive staging environments, yes! Add HTTP Basic Auth via Hostinger cPanel > Directory Privacy.

## Additional Recommendations for Staging

1. **Use HTTP Basic Authentication**
   - Add via Hostinger cPanel
   - Requires username/password to access
   - Simple but effective

2. **Use a Non-Obvious Subdomain**
   - Instead of `staging.eegnite.com`
   - Use `preview-xyz123.eegnite.com`
   - Harder for bots to discover

3. **Monitor Access Logs**
   - Check who's accessing your staging site
   - Look for unexpected bot traffic
   - Available in Hostinger cPanel

4. **Add noindex to Sitemap**
   - Don't submit staging sitemap to Google Search Console
   - Or don't include a sitemap at all on staging

5. **Use Different Analytics**
   - Consider disabling Google Analytics on staging
   - Or use a separate GA property
   - Prevents production data pollution
